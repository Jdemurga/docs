---
title: "Authentication API"
description: "Register users, issue JWTs, link Sovereign sessions, and update preferences through the Vexel Core Auth module."
---

The Auth module exposes user registration, login, Sovereign Link session binding, and preference updates. Every authenticated endpoint expects a bearer JWT in the `Authorization` header.

## Base path

All endpoints are rooted at `/auth`.

## Endpoints

### Register a user

`POST /auth/signup`

Creates a user record. Passwords are hashed with bcrypt before persistence.

**Body**

```json
{
  "email": "user@example.com",
  "pass": "strong-password",
  "name": "Satoshi"
}
```

**Response**

Returns `201 Created` with an empty body on success. Call `POST /auth/login` to obtain a JWT.

### Log in

`POST /auth/login`

Validates credentials and returns a fresh JWT alongside the user summary.

**Body**

```json
{
  "email": "user@example.com",
  "pass": "strong-password"
}
```

**Response**

```json
{
  "accessToken": "<jwt>",
  "user": {
    "id": "<uuid>",
    "email": "user@example.com",
    "name": "Satoshi",
    "preferredLanguage": "en"
  }
}
```

### Link a Sovereign session

`POST /auth/link` — **JWT required**

Binds a Vexel Horizon browser session to the authenticated Mobile identity. Horizon shows a QR code encoding a `sessionId`; Mobile posts the scanned ID here.

**Body**

```json
{
  "sessionId": "4f2c-9a11"
}
```

**Behavior**

- Core validates the Mobile JWT.
- `EventsGateway.notifySessionLink()` emits the `sessionLinked` event to the Horizon socket mapped to `sessionId`.
- Horizon receives a new token and transitions to Sovereign Mode.

See [Real-time events](/core/api/realtime) for the WebSocket contract.

### Update preferences

`POST /auth/preferences` — **JWT required**

Updates mutable user preferences such as language.

**Body**

```json
{
  "preferredLanguage": "es"
}
```

**Response**

Returns `200 OK` with an empty body on success.

## JWT strategy

Vexel Core uses `passport-jwt` with the `JWT_SECRET` environment variable. Include the token on every protected request:

```http
Authorization: Bearer <accessToken>
```

<Info>
  Tokens carry the user ID (as `sub`) and `email` claims. Controllers guarded with `@UseGuards(AuthGuard('jwt'))` receive the decoded payload on `req.user` (mapped to `{ userId, email }` by `JwtStrategy`).
</Info>

## Error responses

| Status | Meaning |
|---|---|
| `400` | Malformed body or missing fields |
| `401` | Invalid credentials or expired token |
| `409` | Email already registered during signup |
| `500` | Unexpected server error — check Core logs |
